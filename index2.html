Introduction
Belkhedma Platform
A unified marketplace aggregating home-service providers — hourly labor, monthly contracts, medical staff, and mediation — into a single app, with payments held in escrow and released on delivery.
The problem: Each provider company has its own CRM, web app, and mobile app. Users must download multiple apps and cannot compare availability or pricing in one place.

Provider integration modes
First Party
Our own labor pool in the Belkhedma CRM org. Always pinned to rank 1, priced with an 8% undercut on competitor prices.
API Mode
Provider exposes their own REST API. We poll it hourly via a background job and normalize the response into our unified ProviderPackage model.
Push Mode
Provider sends data to our webhook endpoint. We validate and normalize on receipt. Same model, same tables as API mode.

API Mode — how it works
1
Trigger

Hangfire/Quartz fires a recurring job every hour for each API-mode provider.

2
ProviderApiAdapter calls provider REST

Hits GET /packages and GET /availability using stored ApiBaseUrl and decrypted ApiKeyHash.

3
Response normalizer

Maps their schema to our internal ProviderPackage model. Stores ExternalPackageRef to correlate back when confirming.

4
Write to AvailabilitySnapshot

Upserts rows keyed by (ProviderPackageId, Date). Sets Source = "Api", updates SyncedAt.

5
Invalidate Redis

Flushes relevant cache keys so the next search picks up fresh data within seconds.

Push Mode — how it works
1
Provider posts to our webhook

POST /api/providers/push/packages — authenticated via HMAC WebhookSecret.

2
Signature verification

We verify the HMAC header before touching the payload. Invalid signatures return 401 immediately.

3
Push normalizer

Same normalization pipeline as API mode — maps to ProviderPackage. Sets Source = "Push".

4
Write & cache invalidate

Same upsert logic. Push-mode providers get faster freshness because they control timing.

Database Domains
The Belkhedma CRM org is split into three logical domains. All tables live in this org under the new ASP.NET Zero tenant.
Domain 01
Providers, Catalog & Entity Mapping
Who offers services
Defines providers, their coverage, service type mapping, and the entity alias system for resolving mismatched reference data.
Provider
Core company record. IntegrationMode = Api | Push. Holds CommissionRate, SponsoredTier, PricingDelegated (bool).
ProviderServiceConfig
Which service groups each provider supports. Holds ExternalGroupCode for API mapping.
ProviderCoverage
FK → ProviderServiceConfig (not Provider directly). City + district per service config. Enables per-service regional control.
ServiceSubTypeKeyword
Canonical keyword vocabulary. FK → ServiceSubType. One agreed slug per subtype (e.g. cleaning, maintenance). Pre-agreed with providers. Sync calls GET /{keyword}/packages — zero per-provider ID mapping.
ServiceGroup
Parent sectors: Hourly (1), Monthly (2), Medical (3), Mediation (4). Self-relation via nullable ParentId — a group can contain sub-groups. Providers never write here.
ServiceSubType
Our canonical sub-services. Each linked to one ServiceSubTypeKeyword slug. Keyword-compliant providers need zero mapping configuration.
EntityKeyword
Keyword bank per canonical entity. Each entity has multiple keywords (e.g. ['egy','egypt','egyptian','مصر','مصري']). Feeds the 3-tier matching engine.
EntityAlias
Resolution cache + review queue. MatchMethod = KeywordExact | SimilarityAuto | ManualReview. Once resolved, future occurrences auto-match from cache without re-running the engine.
City / District / Nationality
Our canonical reference catalogs. Each linked to keywords via EntityKeyword. Never modified by provider data.
ProviderMappingRule
Dynamic JSON/XML path → SQL property mapping rules per provider per service group. Code-less integrations.

Data Model
Mapping Pipeline — Full Journey
How raw provider data becomes a unified, searchable result. Covers four mapping layers.
Layer 1 — Entity Resolution Engine (Cities, Districts, Nationalities)
The problem: Provider A says name='Egypt'. Provider B says name='Egyptian'. Our canonical entity is 'Egyptian'. Same entity, different representations.
Solution: 3-tier matching engine — keyword exact → similarity scoring → manual review
Tier 1 — Cache hit (EntityAlias)
Check EntityAlias for this ProviderId + ExternalId/ExternalName. If found → return CanonicalId instantly. Zero computation. Cache is populated by tiers 2 and 3 — repeat occurrences are always free.
Tier 2 — Keyword + Similarity scoring (EntityKeyword)
Each canonical entity has pre-configured keywords in EntityKeyword. Engine normalizes the incoming name (lowercase, strip diacritics) then:
Keyword exact match
Is normalized name in keyword list? e.g. 'egypt' ∈ keywords. → Auto-match, MatchMethod=KeywordExact. Write to cache.
Similarity ≥ threshold
Jaro-Winkler vs each keyword. ≥ 0.90 → Auto-match SimilarityAuto. 0.70–0.89 → Match + SimilarityFlagged for admin review (still searchable). <0.70 → Unresolved.
Tier 3 — Manual review queue
Similarity <0.70. EntityAlias row with Status=PendingReview. Package excluded from search. Admin maps to correct canonical entity → all packages with that alias resolve instantly.

Layer 4 — Overload Pricing Engine
Two triggers: (1) Provider delegates pricing entirely (PricingDelegated=true). (2) Provider sets price, but we apply platform-level markup or commission baking on top.
Layer A — Provider Base Price
Stored in ProviderBasePrice. If PricingDelegated=true, this is null — overload engine owns the entire calculation.
Layer B — OverloadPricingRule (platform rules)
RuleType: PercentMarkup, FlatFee, CommissionBake, or DelegatedFormula. Scope: Global | PerProvider | PerServiceGroup.
Layer C — VAT Application
Result = LayerBPrice × (1 + VatRate). VatRate = 0 if already VAT-inclusive.
Layer D — FinalDisplayPrice written
Written to FinalDisplayPrice on the detail row. The only price the client ever sees.

Sponsored → completion rate → availability → price → user affinity → rating.

4

Flows
Circuit Breaker & Auto-Lead
Circuit breaker lives entirely in Redis. Every failure also creates a persistent ProviderLeadRequest DB record.
Why Redis and not a DB table?
DB table — wrong tool
DB round-trip on every contract attempt — adds latency.
TTL requires a scheduled cleanup job.
Overkill for ephemeral runtime state — no persistence needed.
Redis — right tool
Sub-millisecond read/write — no latency impact.
Native EXPIRE handles TTL — zero cleanup code.
Atomic INCR — no race conditions on concurrent increments.
Sync job calls DEL to clear instantly on availability change.
Redis key design
Key pattern
cb:{providerId}:{packageId}:{startDate}:{selectedDaysHash}
Example
cb:prov-a3f:pkg-7b2:2025-09-01:sun-mon-wed
Value
Integer — failure count. When ≥ 5, circuit is considered open.
TTL
Set via EXPIRE on first failure. Configurable per provider or globally (default 1 hour).
GET cb:key
Read failure count. ≥ 5 → open → block. Key missing → closed → proceed.
INCR cb:key
Atomic increment per failure. On first failure (INCR returns 1), also call EXPIRE cb:key {ttl}.
DEL cb:key
Called by sync job when package becomes available. Also on successful contract creation.
EXPIRE (auto)
Redis auto-evicts after TTL. Next attempt starts fresh from 0.
Circuit breaker lifecycle
1
User attempts contract creation

Redis GET cb:key. Value ≥ 5 → block, inform user. Key missing or <5 → proceed.

2
Call provider's contract API

Send booking details to provider.

3a
✓ Success

Contract created. DEL cb:key — reset circuit breaker. No ProviderLeadRequest on success.

3b
✗ Failure

Redis INCR cb:key. If result = 1 → also EXPIRE cb:key {ttl}. If result ≥ 5 → circuit now open.

4
Auto-create ProviderLeadRequest (every failure)

On every failure: write ProviderLeadRequest DB record, send to provider API. This is the persistent business record — the Redis key is ephemeral.

5
Reset paths

TTL expiry (Redis auto-evicts) OR sync calls DEL cb:key — whichever comes first.
